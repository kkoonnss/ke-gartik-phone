# v0.5 Plan — Classic Mode Fixes + Reveal Navigation

Source: Kons's notes after live-testing v0.4. This plan is the brief for the next build session. Keep changes surgical; the architecture is sound.

Live URL: https://ke-gartik-phone.onrender.com
Repo: https://github.com/kkoonnss/ke-gartik-phone (public). Deploy = run `push-to-github.bat`, then Render → ke-gartik-phone → Manual Deploy → "Deploy latest commit".

---

## Kons's notes (verbatim intent)

1. In Classic, everyone should be drawing/active at the same time — no waiting for turns.
2. When everyone submits before the timer ends, the round should advance immediately.
3. The reveal "PREV" button does not work.
4. Classic should be a game of telephone: write a prompt → next person draws it → next person describes the drawing → next draws → … so there are as many rounds and as many albums as there are players.

---

## Findings (what's already true vs. what's actually broken)

### Note 1 & 4 — ALREADY IMPLEMENTED (verify only)
`server/modes/classic.js` already implements the exact telephone structure Kons describes:
- `nextPhase`: write(0) → draw(1) → describe(2) → draw(3) → describe(4) → … until `round >= N-1`, then reveal.
- `buildAlbums`: N albums, N slides each. Album j slide i authored by player `(j + i) mod N` (each album passes through every player once and returns to start — true telephone).
- `assignmentForPlayer`: during each phase, every player is assigned a DIFFERENT album's chain, so all players draw (or describe) simultaneously. Nobody waits for a turn.

Conclusion: the "not everyone drawing at once" symptom was almost certainly the pre-v0.4 host-not-playing bug — the host occupied a seat but never drew, so one album-chain stalled and the round ran to full timeout. v0.4 made the host a real player. **Action: verify in a live 3-player test (host on computer + 2 phones) that all three draw at once and the album count = player count.** No code change expected; if verification fails, investigate `emitAssignments` + `getSeatOrder`.

### Note 2 — Early advance EXISTS (verify it now works post-v0.4)
`checkAllSubmitted` in `server/game.js` advances the phase the moment every CONNECTED player has submitted. Pre-v0.4 the host never submitted, so this never fired and every round waited for the timer. With v0.4 host-as-player, the host now submits and counts. **Action: verify a round ends early when all players (including host) submit before the timer.** No code change expected; if it still waits, confirm the host's `phase:submit` is being received and that `connectedPlayers()` matches the submitters.

### Note 3 — PREV button BROKEN (real bug, clear root cause)
Root cause in `server/game.js`:
- `reveal:next` handler guard (~line 691): `if (!room || room.state !== 'reveal') return;`
- `reveal:prev` handler guard (~line 744): `if (!room || room.state !== 'reveal') return;`
- When `reveal:next` advances past the final slide of the final album, it sets `room.state = 'ended'` (~lines 711, 731).

So once the host clicks NEXT to the end, the room is `'ended'`, and from then on BOTH next and prev do nothing (their guard requires exactly `'reveal'`). In a small game the host reaches the end in a few clicks, then "review" via PREV is dead. This matches Kons's report.

---

## The fix (v0.5)

### Fix A — Reveal navigation works in 'ended' too (server/game.js)
1. In BOTH `reveal:next` and `reveal:prev` handlers, change the guard from:
   ```js
   if (!room || room.state !== 'reveal') return;
   ```
   to:
   ```js
   if (!room || (room.state !== 'reveal' && room.state !== 'ended')) return;
   ```
2. In `reveal:prev`, when navigating backward while `room.state === 'ended'`, set `room.state = 'reveal'` before moving the cursor, so the UI re-enters review mode and the reveal panel shows the slide again.
3. Keep the existing behavior where `reveal:next` at the very last slide sets `'ended'` (that's the natural finish). But because prev now works from `'ended'`, the host can always step back to review.
4. Make sure `emitRevealForLayout` / `emitRevealSlide` still fire after a prev from `'ended'` so the slide re-renders on the host screen and the standalone album page.

Edge cases to handle:
- Pressing PREV at album 0 / slide 0: clamp, do nothing (already handled).
- Non-stepper layouts (frame-cycle, gallery, scrollback) reach `'ended'` differently (album-by-album). Apply the same guard relaxation there so album-level prev works after the end.
- After re-entering `'reveal'` via prev, pressing NEXT to the end again should still set `'ended'` cleanly.

### Fix B — Verification harness (no behavior change)
Add a short note to `tests/smoke.js` (or a new manual checklist) that asserts: for N=3 classic, albums.length === 3, each album length === 3, and reveal:prev after reveal-to-end returns state to 'reveal'. The smoke tester already walks reveal:next a few times — extend it to also walk reveal:prev from the ended state and assert no crash + state transitions back to 'reveal'.

### Fix C — (Optional polish, only if verification shows it) 
If the live test shows the round still doesn't end early with the host playing, investigate whether the host's two sockets (host.js + play.js share the singleton, but the server stores one socketId per player) cause the host's submission to be attributed correctly. Expected: fine, because phase:submit looks up the player by the submitting socket's id and both host scripts use the same socket. Only dig here if the verification fails.

## Files in scope
- `server/game.js` — reveal:next / reveal:prev guard + prev-from-ended state restore (Fix A). This is the only required code change.
- `tests/smoke.js` — extend reveal navigation assertions (Fix B).
- No client changes expected — `album.js` already wires the PREV button to emit `reveal:prev`; the bug is purely the server guard.

## Acceptance criteria
1. During reveal, PREV steps backward slide-by-slide (stepper) or album-by-album (other layouts).
2. After clicking NEXT to the very end (state 'ended'), PREV still works and re-enters review.
3. A 3-player Classic game produces 3 albums of 3 slides; all 3 players draw/describe simultaneously each round.
4. A round advances immediately once all players (including the host) submit, without waiting for the timer.
5. Phone players and host page both reflect reveal navigation correctly.
